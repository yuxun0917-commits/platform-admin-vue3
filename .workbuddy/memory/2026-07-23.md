## 强制改密弹窗（用户新需求，推翻上一条「别管 20006」）
- 用户新需求：登录后 `/user/info` 返回 `20006`（新用户需改密码，后端 `ErrorCode.NEED_CHANGE_PWD=20006`）时，**在系统首页（仪表盘）弹出改密弹窗**，而不是登出。
- 后端机制（platform-parent）：`ForceChangePwdInterceptor` 对已登录用户 `password_update_time` 为 null（新用户）或超期时，对所有请求抛 20006 拦截；`/user/reset-pwd` 是自助改密接口（`@SaCheckLogin`，免 `system:user:changePassword` 权限、免二次认证，用户ID取自登录态），改密后写 `SESSION_PWD_UPDATE_KEY` 解除拦截。
- **关键后端修复**：`SaTokenConfig.java` 的 `ForceChangePwdInterceptor` 原本 `addPathPatterns("/**")` 无 exclude，会连 `/user/reset-pwd` 一起拦截 → 前端弹窗会死循环。已加 `.excludePathPatterns("/user/reset-pwd")`。**需重启/重新编译 platform-parent 才生效**。
- 前端落地：
  - `.env` 新增 `VITE_SERVICE_FORCE_CHANGE_PWD_CODES=20006`；`vite-env.d.ts` 加对应类型。
  - `src/service/request/index.ts` 的 `onBackendFail` 新增 `isForceChangePwdCode` 分支：**返回 response**（与 second-factor 同模式），使 caller 能按 `response.data.code` 分支，且不登出、不弹错误 toast。
  - `src/store/modules/auth/index.ts`：`getUserInfo` 识别 `response.data.code === 20006` → 置 `needChangePassword=true` 返回 false；`initUserInfo` 仅当 `!needChangePassword` 时才 `resetStore`；新增 `finishForceChangePwd()`（清除标记 + `window.location.reload()` 全量刷新以干净恢复菜单/数据）。
  - 新增 `fetchUserResetPwd`（`POST /user/reset-pwd`，`src/service/api/user.ts`）。
  - 新增弹窗 `src/views/_builtin/force-change-pwd-modal.vue`：非关闭（closable/maskClosable/keyboard=false、无取消按钮），旧/新/确认密码 RSA 加密后调 `fetchUserResetPwd`，成功 `await authStore.finishForceChangePwd()`。挂载于 `src/App.vue`。
  - 仪表盘各卡片（stat-cards 等）用 `?.` + `?? 0` 兜底，20006 期间不会崩。
- typecheck 通过（exit 0）。

## 强制改密期间侧边栏出现「页面建设中」菜单
- 现象：新用户登录弹改密框时，侧边栏顶部出现一条「页面建设中」菜单（红框所示），排在「仪表盘」前面。
- 根因：**不是 bug**。`src/router/elegant/routes.ts` 的静态路由里本来就有一条 `coming-soon`（i18nKey `route.coming-soon` → 中文「页面建设中」）。正常登录后 `initBackendMenu()` 会用后端 `/user/info` 返回的菜单覆盖静态菜单，因此看不到它；但 20006 强制改密时 `/user/info` 只返回 code 没有 menus，`initBackendMenu()` 直接 return，`menus.value` 仍保留 `initStaticAuthRoute()` 生成的静态菜单，于是 `coming-soon` 暴露出来。
- 修复：`src/store/modules/route/index.ts` 的 `initBackendMenu()` 在「后端菜单为空且处于强制改密状态」时，把 `menus.value` 重置为 `[buildDashboardMenu()]`，只保留仪表盘，不再暴露静态占位菜单。
- typecheck 通过（exit 0）。
